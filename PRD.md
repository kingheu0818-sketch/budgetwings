## 1. 产品愿景 & 要解决的问题

### 1.1 痛点

1. **信息碎片化**：低价机票散落在航司官网、OTA、信用卡积分兑换、各种 Telegram/微信群里，用户需要同时盯十几个渠道。
2. **时效性差**：好价往往几小时内消失，等用户看到攻略时票已售罄。
3. **攻略与票价脱节**：传统旅游攻略不会告诉你"这周末飞大阪往返 ¥800"，而特价信息也不会告诉你到了目的地怎么玩。
4. **人群需求差异大**：打工人只有周末/小长假，预算相对宽裕但时间紧迫；学生党时间灵活但预算极低——现有工具不做区分。

### 1.2 愿景

构建一个**开源、每日自动更新**的出行情报系统，做到：
- 自动抓取 & 聚合多渠道低价出行信息（机票、火车、大巴、拼车）
- 根据当前可用低价票，**反向生成短期旅游攻略**（"票价驱动型攻略"）
- 针对**打工人**和**学生党**两种用户画像，输出不同策略的推荐

### 1.3 对标 & 差异化

| 对标产品 | 它做了什么 | BudgetWings 的差异 |
|----------|-----------|-------------------|
| OpenClaw | 低价机票聚合 | 我们在票价基础上叠加攻略生成 |
| 去哪儿/飞猪 | OTA 比价 | 我们面向极致低价 + 开源社区驱动 |
| 穷游/马蜂窝 | 攻略社区 | 我们的攻略是由实时票价驱动的，不是静态内容 |
| Google Flights | 价格追踪 | 我们聚焦亚洲市场 + 人群策略 + 多交通方式 |

## 2. 用户画像 & 场景

### 2.1 打工人画像

| 维度 | 描述 |
|------|------|
| 年龄 | 22-35 岁 |
| 时间约束 | 周末（2天）、小长假（3-5天）、年假（拼假7-10天） |
| 预算区间 | 单程机票 ¥300-1500，整趟旅行 ¥1000-5000 |
| 出发时间灵活度 | 低。需要精确到具体周末 |
| 核心诉求 | "这个周末能去哪？帮我直接安排好" |
| 额外需求 | 信用卡积分兑换建议、航司里程玩法、拼假日历 |

### 2.2 学生党画像

| 维度 | 描述 |
|------|------|
| 年龄 | 18-25 岁 |
| 时间约束 | 寒暑假（长线）、周末+逃课（短线）、考试周后 |
| 预算区间 | 单程机票 ¥100-500，整趟旅行 ¥200-2000 |
| 出发时间灵活度 | 高。可以为了低价改日期 |
| 核心诉求 | "最近哪里最便宜？我时间随便" |
| 额外需求 | 青旅/沙发客推荐、学生票优惠、夜间巴士/火车卧铺省住宿 |

### 2.3 核心用户场景

**场景 A：打工人的周五晚上**
> 周五下班，刷到推送："本周末深圳→清迈 往返 ¥680，含 2 天 1 夜速游攻略"。点进去看到行程安排、签证提示（落地签）、当地必去的 3 个点、预算明细。直接订票出发。

**场景 B：学生的暑假选择困难症**
> 暑假想出去浪但没钱。打开 BudgetWings，选"学生模式"，看到"本周最低价 TOP 10"：昆明→西双版纳 ¥99 火车硬座、上海→济州岛 ¥199 特价机票……每个目的地都配了穷游版攻略，标注了免费景点和 ¥30 以下的住宿。

**场景 C：社区贡献者**
> 一位经常薅机票羊毛的达人，把自己发现的 bug fare（错误票价）通过 PR 提交到数据源，或者为某个目的地补充了攻略模板。

## 4. 双模式策略设计——"打工人" vs "学生党"

这是产品的核心差异化，不是简单的 UI 换肤，而是**底层筛选逻辑和内容生成策略的不同**。

### 4.1 筛选逻辑差异

| 维度 | 打工人模式 | 学生党模式 |
|------|-----------|-----------|
| 默认日期范围 | 未来 2-4 个周末 + 下一个小长假 | 未来 30-90 天（越灵活越便宜） |
| 价格上限（单程） | ¥1500 | ¥500 |
| 交通方式优先级 | 飞机 > 高铁 > 其他 | 火车硬座/硬卧 > 廉航 > 大巴 |
| 出发时间偏好 | 周五晚 / 周六早 出发，周日晚返回 | 不限，哪天便宜飞哪天 |
| 是否含红眼航班 | 默认排除，可手动开启 | 默认包含（省一晚住宿） |
| 中转容忍度 | 仅直飞或短中转（<3h） | 可接受长中转（甚至当中转站玩一天） |

### 4.2 攻略内容差异

| 内容模块 | 打工人版本 | 学生党版本 |
|----------|-----------|-----------|
| 行程节奏 | 紧凑高效，精选必去 3-5 个点 | 慢节奏，可深度探索 |
| 住宿推荐 | 经济型酒店 / 民宿（¥200-500/晚） | 青旅/胶囊/沙发客（¥30-100/晚） |
| 餐饮建议 | 本地人气餐厅（人均 ¥50-150） | 街边摊 / 夜市 / 便利店攻略（人均 < ¥30） |
| 交通方式 | 打车/包车/地铁 | 公交/步行/共享单车/搭车 |
| 预算明细 | 含打车、正餐、门票 | 列出所有免费景点，标注学生票折扣 |
| 特色模块 | "请假邮件模板"、"拼假攻略" | "穷游生存指南"、"免费 Wi-Fi 地图" |

### 4.3 展示差异

| 元素 | 打工人模式 | 学生党模式 |
|------|-----------|-----------|
| 首页标题 | "这个周末去哪浪？" | "最便宜能去哪？" |
| 排序默认值 | 按日期（最近的周末优先） | 按价格（最便宜优先） |
| 预算标签 | 🏷️ "周末微旅行 ¥2000 以内" | 🏷️ "百元穷游挑战" |
| 额外信息 | 航司里程累积提示 | 学生证优惠提示 |

## 5. 攻略生成引擎设计

### 5.1 "票价驱动型攻略"的生成逻辑

传统攻略：先选目的地 → 再找票。
BudgetWings：先发现低价票 → 反向生成攻略。

```
低价票数据进入
      │
      ▼
┌─────────────┐
│ 目的地识别   │ ← 从票价数据提取目的地城市
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 攻略模板匹配 │ ← 从模板库中匹配该城市的攻略骨架
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 动态数据注入 │ ← 注入实时票价、天气、签证政策、汇率等
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 人群策略适配 │ ← 根据打工人/学生党模式调整内容
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 输出攻略卡片 │ ← 生成结构化攻略内容
└─────────────┘
```

## 6. 数据采集策略

### 6.1 数据源优先级

| 优先级 | 数据源类型 | 示例 | 备注 |
|--------|-----------|------|------|
| P0 | 航司官方 API/数据 | 各廉航官网 | 需要注意反爬和 ToS |
| P0 | 公开比价 API | Skyscanner Affiliates API, Kiwi.com API | 有免费 tier |
| P1 | OTA 数据 | 去哪儿、飞猪、携程 | 反爬风险高，需要代理池 |
| P1 | 铁路数据 | 12306 | 有第三方 API 可用 |
| P2 | 社区提交 | GitHub Issues / PR | Bug Fare、限时促销等 |
| P2 | 航司促销页 | 各航司官网促销专区 | 结构化程度低，需定制解析 |

### 6.2 采集架构建议

- **定时任务**：GitHub Actions (免费) 或 cron job，每天 2-4 次
- **数据存储**：SQLite（MVP 阶段）→ PostgreSQL（扩展阶段）
- **缓存**：文件系统 JSON 缓存 → Redis
- **输出格式**：JSON + Markdown（方便静态站点生成）

### 6.3 合规 & 伦理红线

- ❌ 不做购票代理，不经手任何交易
- ❌ 不存储用户支付信息
- ✅ 只做信息聚合和导流，用户点击后跳转到官方渠道购票
- ✅ 明确标注数据来源和抓取时间
- ⚠️ 遵守各数据源的 robots.txt 和 ToS
- ⚠️ 设置合理的爬取频率，不给源站造成压力

## 8. 我补充的想法（你没提到但值得做的）

### 8.1 "中转城市也能玩" 功能
很多低价票是需要中转的。与其把中转当缺点，不如反过来做一个"中转城市速玩攻略"。比如在吉隆坡中转 8 小时，自动生成一个"8 小时吉隆坡快闪攻略"，告诉用户出不出关、怎么快速往返市区、吃什么。

### 8.2 "出行搭子" 匹配（社区功能）
发现同一航班/同一目的地的用户可以选择匹配同行伙伴。对学生党尤其有吸引力——拼房省住宿费。不过 MVP 阶段可以简化为 Telegram 群组按目的地自动建群。

### 8.3 "逆向日历" 视图
不是"我要 X 月 X 日去哪"，而是"未来 60 天，每个周末最便宜能去哪"的日历热力图。用户一眼看出哪个周末有好价。

### 8.4 "预算倒推" 模式
用户输入总预算（比如 ¥2000），系统反向推荐"这个预算够你去哪"，包含机票+住宿+吃喝的完整估算。

### 8.5 "签证友好度" 筛选
自动标注免签/落地签/电子签/需要提前办理，学生党模式下优先推荐免签和落地签目的地（因为办签证也是一笔开销和时间成本）。

### 8.6 "薅羊毛指南" 模块
不只是找低价票，还教用户怎么薅：
- 各航司会员日日历（南航 28 号、东航 18 号等）
- 信用卡航空联名卡新户优惠汇总
- 各 OTA 平台新人券/学生认证优惠
- 里程兑换入门教程

### 8.7 "实时特价告警" 机制
除了每日更新外，对于真正的 Bug Fare / Flash Sale，需要分钟级推送。可以设计一个社区驱动的速报系统：发现者提交 → 自动验证票价 → 群发通知。

### 8.8 开源社区运营策略
- 攻略模板用 YAML 存放，降低贡献门槛（不需要会写代码也能贡献攻略）
- 设计 "Good First Issue" 标签体系，引导新人参与
- 每月发布"社区贡献排行榜"
- 攻略模板审核流程：PR → 至少 1 人 review → 合并

## 11. 风险 & 应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 数据源反爬/封 IP | 数据断供 | 优先使用官方 API；备用多个数据源；设置合理频率 |

| 法律合规风险 | 项目下架 | 不做交易、不存隐私数据、标注来源、尊重 ToS |
| 开源项目维护疲劳 | 项目停更 | 尽量自动化、清晰的贡献指南、培养 co-maintainer |

## 12. Codex 开发指引

以下是给 Codex 写代码时的核心原则，也是后续 Code Review 的评审标准：

### 代码质量
- 清晰的模块分离，每个数据源一个独立的采集器类
- 完善的错误处理和重试机制（网络请求必须有 timeout 和 retry）
- 类型注解（Python 用 type hints，TypeScript 用 strict mode）
- 单元测试覆盖率 ≥ 核心逻辑 80%

### 数据标准
- 所有票价数据统一为 JSON Schema，字段命名清晰
- 时间统一用 UTC，前端再做本地化
- 价格统一存 CNY 分（整数），展示时再转换

### 安全
- 不在代码中硬编码任何 API Key，全部用环境变量
- 爬虫的 User-Agent 设置为包含项目名称的友好字符串
- 不存储任何用户个人信息（MVP 阶段）

### 可维护性
- 新增数据源只需要实现一个标准接口，不需要改动核心逻辑
- 攻略模板的 YAML schema 必须有文档说明
- GitHub Actions workflow 需要有失败告警（通知到 Telegram）
